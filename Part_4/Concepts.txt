# ============================================================

# 🔹 WHAT IS HUMAN-IN-THE-LOOP (HITL)?

# ============================================================

# HITL is a mechanism where agent execution is PAUSED before

# running a sensitive or risky tool, waiting for human approval.

#

# Flow:

# Agent runs

# → Tool call generated

# → needs_approval checked

# → ⏸ Execution paused

# → Human approves / rejects

# → Resume execution

#

# Example:

# delete_file("data.db")

# → pause → ask user → resume

# ============================================================

# 🔹 WHY HITL IS IMPORTANT?

# ============================================================

# - Prevents dangerous actions (delete, payments, emails)

# - Adds human control in automation

# - Required in enterprise / compliance systems

# - Enables trust in AI systems

# ============================================================

# 🔹 HOW HITL WORKS INTERNALLY

# ============================================================

# 1. Agent generates tool call

# 2. Runner checks needs_approval

# 3. If approval required:

# → execution pauses

# → result.interruptions populated

# 4. Convert to state:

# state = result.to_state()

# 5. Approve or reject:

# state.approve(interruption)

# state.reject(interruption)

# 6. Resume:

# Runner.run(agent, state)

# ============================================================

# 🔹 WHAT IS result.to_state()?

# ============================================================

# Converts paused execution into a resumable object.

#

# Think of it like:

# → "Save game state"

#

# It stores:

# - current execution step

# - pending tool calls

# - approvals

# - conversation history

#

# Without this → cannot resume execution

# ============================================================

# 🔹 WHAT ARE INTERRUPTIONS?

# ============================================================

# result.interruptions contains pending approvals.

#

# Each interruption has:

# - tool name

# - arguments

# - agent info

#

# You MUST handle them before continuing.

# ============================================================

# 🔹 WHY TOOL DOESN'T ASK AGAIN NEXT TIME?

# ============================================================

# Because of approval memory:

#

# approvals[key] = True

#

# Next time:

# if approvals.get(key):

# return False  # skip approval

#

# Meaning:

# → Already trusted → no need to ask again

# ============================================================

# 🔹 WHY FALSE IS NOT STORED?

# ============================================================

# If user rejects:

# → We DO NOT store it

#

# Because:

# - rejection may change later

# - approval is trust signal

# - rejection is temporary decision

#

# If needed:

# approvals[key] = False  # can be added manually

# ============================================================

# 🔹 CONDITIONAL APPROVAL LOGIC

# ============================================================

# You can control approval using Python logic:

#

# async def needs_approval(ctx, params, id):

# if params["filename"].endswith(".db"):

# return True

# return False

#

# This gives FULL CONTROL over when HITL triggers

# ============================================================

# 🔹 CAN WE INTERRUPT MID-FLOW (NOT JUST TOOL)?

# ============================================================

# YES — but indirectly.

#

# You cannot pause inside Python execution of a tool.

#

# Instead:

# - break flow into steps

# - use tools as checkpoints

#

# Example:

# Step 1 → categorize

# Step 2 → validate_category (HITL here)

# Step 3 → process

#

# So interruption happens BETWEEN steps

# ============================================================

# 🔹 CAN TOOL CALL ANOTHER TOOL?

# ============================================================

# ❌ Not directly (inside Python function)

#

# Tools are atomic

#

# ✅ Instead:

# - return result

#

# Correct flow:

# Agent → Tool A → Agent → Tool B

# ============================================================

# ============================================================
# 🔹 WHAT IS STREAMING?
# ============================================================

# Streaming means the agent returns output in chunks (real-time),
# instead of waiting for the full response.

# Example:
# Normal:
#   "Hello, how can I help you?"
#
# Streaming:
#   H...
#   He...
#   Hello...

# ============================================================
# 🔹 NORMAL RUN vs STREAMING RUN
# ============================================================

# Normal:
#   result = await Runner.run(agent, input)
#   print(result.final_output)
#
# Streaming:
#   result = await Runner.run_streamed(agent, input)
#   async for event in result.stream_events():
#       print(event)

# ============================================================
# 🔹 STREAMING + HITL (IMPORTANT)
# ============================================================

# HITL (Human-in-the-loop) works the SAME in streaming.

# Flow:
#   1. Start streaming run
#   2. Agent starts generating output
#   3. Tool requires approval
#   4. Execution PAUSES
#   5. interruptions appear
#   6. Human approves/rejects
#   7. Resume streaming

# 👉 Key idea:
# Streaming does NOT change HITL logic
# It only changes how output is delivered

# ============================================================
# 🔹 HOW TO HANDLE STREAM EVENTS
# ============================================================

# result = await Runner.run_streamed(agent, input)
#
# async for event in result.stream_events():
#     print(event)   # partial output

# After stream ends:
#   check result.interruptions

# ============================================================
# 🔹 INTERRUPTION HANDLING (SAME AS NORMAL RUN)
# ============================================================

# if result.interruptions:

#
#     for interruption in result.interruptions:
#         state.approve(interruption)  # or reject
#
#     # resume streaming

# 🔹 WHAT IS SESSION?
# ============================================================

# Session = persistent memory stored in database

# It stores:
# ✔ conversation history
# ✔ tool calls
# ✔ agent context

# ============================================================
# 🔹 WITHOUT SESSION
# ============================================================

# Run 1:
#   User: "Hello"
#
# Run 2:
#   Agent forgets everything ❌

# ============================================================
# 🔹 WITH SESSION
# ============================================================

# Run 1:
#   User: "Hello"
#
# Run 2:
#   Agent remembers previous messages ✅

# ============================================================
# 🔹 IMPORTANT RULE (VERY CRITICAL)
# ============================================================

# When resuming after interruption:
# 👉 ALWAYS pass SAME session

# ❌ WRONG:
# result = await Runner.run(agent, state)
#
# ✔ Agent loses memory

# ✅ CORRECT:
# result = await Runner.run(agent, state, session=session)

# ============================================================
# 🔹 STATE vs SESSION (VERY IMPORTANT)
# ============================================================

# State:
#   - Execution snapshot
#   - Where agent paused
#   - Used for resuming

# Session:
#   - Conversation memory
#   - Stored in DB
#   - Used across multiple runs

# ============================================================
# 🔥 COMBINED UNDERSTANDING
# ============================================================

# Session → remembers conversation
# State   → remembers execution pause point
